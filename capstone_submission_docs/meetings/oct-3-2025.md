### **1. Key Discussion Points**

#### **1.1. High-Level System Architecture**

The team conducted a whiteboarding session to outline the primary components of the system. The proposed architecture consists of:

- **Front-End:** The client-side application, which will include the Integrated Development Environment (IDE) for students and a separate interface for instructors/TAs.
- **API/Backend:** The central service layer responsible for handling data, business logic, and communication between the front-end and other components.
- **Database:** The persistence layer for all project data.
- **Code Runner:** An isolated environment for executing user-submitted code.
- **Authentication Service:** Manages user identity and access control.

#### **1.2. API Requirements & Design**

The team identified several critical requirements for the API:

- **Event Ingestion:** The API must be capable of handling a high volume of events from the front-end to capture granular user interactions. This includes keystrokes, copy-paste events, focus changes (e.g., tab switching), and code execution triggers.
- **Real-Time Communication:** A real-time data layer is essential for the live monitoring feature. **WebSockets** was identified as the preferred protocol for this functionality.
- **Code Execution Endpoint:** A dedicated endpoint will be required to interface with the Code Runner service.
- **Metadata Management:** Standard RESTful endpoints will be needed for managing non-real-time data such as user profiles, project details, and submissions.

#### **1.3. Database and Data Model**

The core data entities to be stored were defined as:

- **User Data:** Authentication (AuthN) and Authorization (AuthZ) information, including roles (student, TA, instructor).
- **Code Metadata:** Information linking code to a specific user, project, and submission. This will include submission state (e.g., draft, submitted).
- **Version History:** A highly granular, event-driven history of all changes made to a document or code file. This is crucial for both the playback feature and the cheating detection analysis.

#### **1.4. Front-End Technology Stack**

- **Framework:** After discussion, **Next.js** was selected as the front-end framework. The team agreed to use the **Pages Router** for its stability and better alignment with the application's client-side-heavy requirements.
- **IDE Component:** The **Monaco Editor** library (the engine behind VS Code) was chosen to provide the core code editing experience.
- **Styling:** **Tailwind CSS** will be used for styling. Component libraries like **shadcn/ui** were mentioned as a possibility.

#### **1.5. Code Execution and Authentication**

- **Code Runner:** The team discussed the complexity of building a secure code execution sandbox. To manage scope, using a third-party service is the preferred approach. **E2B** was mentioned as a potential candidate.
- **Authentication:** A dedicated library will be used to handle authentication. **BetterAuth** was proposed as the solution to integrate with the backend.

### **2. Decisions Made**

1.  **Backend Framework:** The team has decided to proceed with **Convex** as the backend framework. Its integrated real-time database and API capabilities are well-suited for the project's requirements and will handle the complexities of WebSockets automatically.
2.  **Front-End Framework:** **Next.js (with Pages Router)** is the confirmed choice for the front-end application.
3.  **Authentication:** The project will utilize the **BetterAuth** library for managing user authentication and roles.
